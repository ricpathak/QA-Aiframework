# Generate Playwright Tests from Source Code

Analyze source code and generate Playwright automated test cases that follow this project's
conventions (fixtures, page objects, `@Step()` decorator, `TAGS` enum, shared mock-data).

## Your task

$ARGUMENTS

If no arguments were given, ask the user:
1. Which source file(s) or directory to analyze (e.g. `src/components/Login.tsx`, `src/api/orders.ts`)
2. Whether there is a live URL to test against (optional — needed for browser interaction steps)

---

## Step 1 — Read framework conventions

Read these files to understand the project's rules before generating anything:

- `utils/configuration.ts` — TAGS enum, environments
- `utils/step-decorator.ts` — how @Step() works
- `src/pages/ui/base-page.ts` — BasePage methods available to all pages
- `src/pages/ui/todo-page.ts` — reference page object (shows the correct style)
- `src/tests/ui/fixtures/todo-fixture.ts` — reference fixture (shows the correct style)
- `src/shared/mock-data/` — what test data already exists
- `src/tests/ui/todo.spec.ts` — reference spec (shows the correct test structure)

## Step 2 — Analyze the source code

Read the files the user pointed at. Follow imports one level deep to understand:
- What the feature/component/service does
- What inputs it accepts and what state changes it produces
- What error states and edge cases exist
- What API endpoints are involved (if any)

Also check:
- `src/pages/ui/` — does a page object already exist for this feature?
- `src/tests/ui/generated/` — are any scenarios already covered?
- `requirements/` — are there requirement IDs (REQ-NNN) to link?

Summarize what you found in 3–5 bullet points before writing any code.

---

## Step 3 — Derive test scenarios

From the source code, identify:

**Smoke tests** (critical paths):
- The primary happy-path user flow
- Any authentication or access boundary

**Regression tests** (edge cases):
- Invalid inputs and validation errors
- Boundary values (empty, too long, special chars)
- Error states (API failures, locked accounts, empty lists)
- State-dependent behavior (disabled buttons, conditional rendering)

List all scenarios before writing any file. One line each:
```
[SMOKE]      Valid login → lands on dashboard
[REGRESSION] Wrong password → error message shown
[REGRESSION] Empty email → validation error
[REGRESSION] Locked account → locked message shown
```

## Step 4 — Prepare the page object layer

**If a page object already exists** for this feature — read it fully, then add only the
methods you need for the new tests. Apply `@Step()` to every new public method:

```typescript
import { Step } from "@utils/step-decorator";

@Step()
async clickSubmit(): Promise<void> {
  await this.submitButton.click();
}
```

**If no page object exists** — create `src/pages/ui/<feature>-page.ts`:
- Extend `BasePage`
- Define locators in the constructor using `getByTestId` → `getByRole` → `getByLabel` → `getByPlaceholder` (in that preference order, never CSS selectors)
- Apply `@Step()` to every public method
- Export it from `src/pages/ui/index.ts`

**If no fixture exists** — create `src/tests/ui/fixtures/<feature>-fixture.ts`:
- Follow the exact pattern of `todo-fixture.ts`
- Extend `base` from `@playwright/test`
- Instantiate the page object, navigate to the URL, expose via `use()`

---

## Step 5 — Write the spec files

Create one file per scenario under `src/tests/ui/generated/`.

**Required header** (every file must have this):
```typescript
// @generated by playwright-source-code-test-generator
// source: <comma-separated source files you analyzed>
// scenario: <scenario title>
// req: <REQ-NNN if found, otherwise "src-derived">
```

**Required structure:**
```typescript
import { test } from "../fixtures/<feature>-fixture";
import { TAGS } from "@utils/configuration";
import { FEATURE_DATA } from "shared/mock-data";

test.describe("<Feature name>", () => {
  test(
    "<scenario title>",
    { tag: [TAGS.SMOKE] },
    async ({ featurePage }) => {

      await test.step("<what you're doing>", async () => {
        await featurePage.someAction();
      });

      await test.step("<what you're asserting>", async () => {
        await featurePage.expectSomethingVisible();
      });
    },
  );
});
```

**Non-negotiable rules:**
- Import `test` from the **fixture file**, never from `@playwright/test`
- All locator interactions go through **page object methods** — no raw `page.*` in spec files
- Every logical action and assertion must be in a `test.step()`
- Tags must use `TAGS.SMOKE` or `TAGS.REGRESSION` from `@utils/configuration`
- Test data must come from `shared/mock-data` — never hardcoded strings
- No `waitForTimeout`, no `networkidle`, no `setTimeout`
- Every spec needs a `test.describe()` block
- Credentials always from `process.env.APP_USERNAME` / `process.env.APP_PASSWORD` — never hardcoded

## Step 6 — Validate

After writing each file, check it passes these rules (from `utils/validate-generated-tests.ts`):

- [ ] Has `// @generated by` header
- [ ] Has `// spec:` or `// source:` reference
- [ ] Has `// scenario:` reference
- [ ] Has `// req:` reference
- [ ] Has `test.describe()` block
- [ ] Has `expect()` or page object `expect*()` calls
- [ ] No `waitForTimeout` / `networkidle` / `setTimeout`
- [ ] No commented-out `expect()` calls
- [ ] No `test.only()`

If anything is missing, fix it before moving to the next scenario.
