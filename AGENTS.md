# Playwright Test Agents — Architecture & Governance

This document defines the contract, responsibilities, and maintenance rules for the four agents in the AI-driven QA pipeline.

---

## Agent Pipeline Overview

```
Orchestrator (HITL checkpoints between phases)
    ├─ Phase 1 → Planner (explores app, generates test plan)
    ├─ Phase 2 → Generator (turns plan into executable tests)
    ├─ Phase 3 → Executor (runs tests, collects failures)
    └─ Phase 4 → Healer (debugs failures, patches tests)

Source Code Test Generator (standalone — no test plan required)
    ├─ Phase 0 → Load framework conventions
    ├─ Phase 1 → Analyze source code + derive scenarios
    ├─ Phase 2 → Prepare page objects / fixtures
    ├─ Phase 3 → Generate spec files
    ├─ Phase 4 → Validate every file
    └─ Phase 5 → Report
```

Each agent is defined in `.github/agents/<name>.agent.md` with:

- YAML frontmatter: tools, MCP servers, model
- System prompt: role, workflow, rules

## Agent Roles & Responsibilities

### 1. Planner — `playwright-test-planner.agent.md`

**Role:** Explore the live app UI and generate a requirements-driven test plan.

**Enforced Conventions:**

- Every scenario must map to a REQ-NNN if a requirement file exists
- Every scenario carries a tag recommendation: `@smoke` (high priority) or `@regression` (medium/low)
- Scenarios reference locator strategy: prefer data-test IDs, then ARIA roles
- Plans are saved to `specs/<feature-name>-plan.md`

**Startup Sequence (non-negotiable):**

1. Call `get_framework_conventions` → load project POM rules
2. Resolve requirements source in priority order: Jira issue key → requirements file → pasted requirements
3. Convert the selected source payload to canonical structure via `normalize_requirements`
4. Call `planner_setup_page` → initialize browser

**Key Rules:**

- Steps must be specific enough for any tester to follow without ambiguity
- Include negative testing: invalid input, empty states, error handling
- Scenarios are independent and runnable in any order
- Flag gaps: requirements with no discoverable UI flow

**Output:** `specs/<feature-name>-plan.md` with clear section headings, numbered steps, tag recommendations

---

### 2. Generator — `playwright-test-generator.agent.md`

**Role:** Convert plan scenarios into executable `.spec.ts` files following project conventions.

**Enforced Conventions:**

1. **Always use the fixture**, never raw `@playwright/test`:

   ```ts
   import { test } from "../fixtures/<app>-fixture";
   ```

2. **Always use page objects**, never raw `page.*` in test bodies:

   ```ts
   await todoPage.addTodo("item"); // ✓
   await page.locator(".new-todo").fill("item"); // ✗
   ```

3. **Wrap steps in `test.step()`**, never inline comments:

   ```ts
   await test.step("Add a todo item", async () => { ... });  // ✓
   // Step 1: Add a todo item  // ✗
   ```

4. **Apply tags** based on planner's recommendation:

   ```ts
   import { TAGS } from "@utils/configuration";
   test("scenario", { tag: [TAGS.SMOKE] }, async ({ todoPage }) => { ... });
   ```

5. **Use shared mock data**, never hardcoded strings:

   ```ts
   import { TODO_ITEMS } from "shared/mock-data";
   await todoPage.addTodo(TODO_ITEMS[0]); // ✓
   ```

6. **File header with provenance & traceability:**
   ```ts
   // @generated by playwright-test-generator
   // spec: specs/todo-test-plan.md
   // scenario: Add Valid Todo
   // req: REQ-001
   ```

**Startup Sequence (non-negotiable):**

1. Read `src/tests/ui/fixtures/` → find fixture for the app
2. Read `src/pages/ui/` → find or create page objects
3. Read `src/shared/mock-data/` → understand available test data
4. Read `src/shared/utils/` → know what helpers exist
5. Read an existing hand-written test → confirm import paths

**Key Rules:**

- One file per scenario, kebab-case name
- One `test.describe` per plan section
- Never use `page.waitForTimeout` or `waitForNetworkIdle`
- Call `generator_setup_page` before each scenario
- Call `generator_write_test` with generated code

**Output:** `src/tests/ui/generated/<scenario-name>.spec.ts` with all conventions applied

### 3. Healer — `playwright-test-healer.agent.md`

**Role:** Debug failing tests and apply minimal, targeted fixes.

**Enforced Conventions:**

- Prefer `getByRole`, `getByLabel`, `getByTestId` over CSS selectors
- Prefer regex matchers for dynamic text
- Never use `waitForNetworkIdle` or `waitForTimeout`

**Startup Sequence (non-negotiable):**

1. Call `get_framework_conventions` with section `"locators"` → know preferred selectors
2. Call `get_test_health` or `list_test_health` → check heal history

**Critical Guard:** **Healing Limit**

- If `healCount >= 3` for a test: **STOP.** Do not heal again.
- Report: "This test has been healed 3+ times. Recommend regenerating instead."
- Rationale: Over-healing compounds fragility; better to regenerate from the plan.

**Healing Workflow:**

1. Run `test_run` → identify failures
2. Call `get_test_failures` → get structured JSON (file, line, error, screenshot)
3. For each failure: run `test_debug` → pause at error
4. Use `browser_snapshot` + `browser_generate_locator` → inspect page state
5. Root cause analysis:
   - Stale selector? → check page object locator strategy first
   - Timing issue? → add Playwright auto-waiting (no manual waits)
   - Assertion mismatch? → **STOP. Do not modify assertions.** Mark `test.fixme()` with evidence, requires human approval
6. **Apply fix based on root cause:**
   - **Locator issues**: Update page object to use preferred strategy (getByRole > getByTestId), never CSS
   - **Timing issues**: Add proper awaits, never `waitForTimeout`
   - **Assertion issues**: Mark `test.fixme()`, document observed vs expected, flag for approval
7. Re-run specific test with `test_run` → verify
8. Call `record_heal_event` → log what was fixed with **required fields**:
   - `healType`: `"selector"` | `"timing"` | `"logic"` (use `"assertion"` ONLY with `humanApprovedAssertionChange=true`)
   - `evidence`: screenshot path or DOM description proving the locator was wrong, not the assertion
   - `productBehaviorChanged`: `false` for selector/timing fixes; `true` if app behaviour differs (triggers product bug flag)
9. Iterate until all pass

**Key Rules:**

- Fix one issue at a time, retest before moving on
- If test fails after 2 fix iterations and logic is correct: mark `test.fixme()` with explanation
- Always call `record_heal_event` after successful fix
- Do not ask user questions — do the most reasonable thing

**Output:** Patched `.spec.ts` file, heal event recorded in `test-health.json`

### 5. Source Code Test Generator — `playwright-source-code-test-generator.agent.md`

**Role:** Analyze existing source code (components, services, API routes, models) and generate thorough, maintainable Playwright test cases — without requiring a pre-written test plan.

**Invocation:** Standalone. Not part of the Orchestrator pipeline. Called directly by a developer who points it at source files.

Also available as a Claude Code skill: `/generate-tests <path>` (`.claude/commands/generate-tests.md`).

**Enforced Conventions:**

1. **Load conventions first** — calls `get_framework_conventions` before any other action; every decision must align with what it returns.

2. **Fixture import** — generated specs always import `test` from the feature fixture, never from `@playwright/test` directly.

3. **Page object methods only** — no raw `page.*` calls in spec files; all interactions go through page object methods.

4. **`@Step()` on every new page object method** — if a page object is created or extended, every public method must be decorated:
   ```ts
   import { Step } from "@utils/step-decorator";

   @Step()
   async clickSubmit(): Promise<void> {
     await this.submitButton.click();
   }
   ```

5. **Locator strategy** — `getByTestId` → `getByRole` → `getByLabel` → `getByPlaceholder`; never CSS selectors or XPath.

6. **Tags from enum** — `TAGS.SMOKE` for happy paths, `TAGS.REGRESSION` for edge cases; never string literals.

7. **Mock data** — test data must come from `src/shared/mock-data/`; generator adds to mock data rather than inlining strings.

8. **No brittle waits** — `waitForTimeout`, `networkidle`, `setTimeout` are forbidden; rely on Playwright auto-wait.

9. **Required file header** (every generated spec):
   ```ts
   // @generated by playwright-source-code-test-generator
   // source: <comma-separated source files analyzed>
   // scenario: <scenario title>
   // req: <REQ-NNN if found in requirements/, otherwise "src-derived">
   ```

10. **`test.describe()` required** — every spec must have a describe block; no flat tests.

11. **Export new page objects** — any new page object must be added to `src/pages/ui/index.ts`.

**Startup Sequence (non-negotiable):**

1. Call `get_framework_conventions` → load POM rules, tagging, fixture patterns
2. Read target source files provided by the user
3. Follow imports one level deep to understand types and service contracts
4. Scan `src/pages/ui/`, `src/tests/ui/fixtures/`, `src/tests/ui/generated/`, `requirements/`
5. Derive scenario list (SMOKE + REGRESSION) before writing any file
6. Prepare page object layer (create or extend)
7. Generate spec files one scenario at a time
8. Call `validate_generated_test` on every file before reporting done

**Key Rules:**

- Never require a pre-written test plan — derives it from source code
- Skips scenarios already covered in `src/tests/ui/generated/`
- Uses `// req: src-derived` when no `requirements/` files are found
- Credentials always via `process.env.APP_USERNAME` / `process.env.APP_PASSWORD` — never hardcoded
- No `test.only()` in committed code
- One spec file per scenario, kebab-case filename

**Output:**
- `src/pages/ui/<feature>-page.ts` (created or updated)
- `src/tests/ui/fixtures/<feature>-fixture.ts` (created if missing)
- `src/tests/ui/generated/<scenario-name>.spec.ts` (one per scenario)

### 4. Orchestrator — `playwright-test-orchestrator.agent.md`

**Role:** Coordinate Planner → Generator → Executor → Healer with human review checkpoints.

**Enforced Conventions:**

- Inherits all conventions from Planner, Generator, and Healer
- Applies them sequentially with **3 human review checkpoints:**

1. **Checkpoint 1** (after Plan):

   - Present plan to user with scenario count and coverage
   - Ask: "Add, remove, or change any scenarios? Confirm Go."
   - Do not proceed until explicit confirmation

2. **Checkpoint 2** (after Generate):

   - List all generated test files with scenario names
   - Ask: "Open any file to review. Should any be skipped? Confirm Go."
   - Do not proceed until explicit confirmation

3. **Checkpoint 3** (after Execute, if failures):
   - Show failure table: test name, file, error, failed step
   - Ask: "Heal these automatically? Confirm Go or Skip."
   - Do not proceed unless explicitly confirmed

**Key Workflow:**

- Phase 1: Run Planner logic → save plan → checkpoint
- Phase 2: Run Generator logic for each scenario → checkpoint
- Phase 3: Run all tests → if all pass, jump to report; if fail → checkpoint
- Phase 4: Run Healer logic on each failure → re-run tests
- Phase 5: Generate final report with coverage & execution summary

**Requirements Intake Modes:**

- Jira issue key (for example, `PROJ-123`) via Jira MCP or Jira API
- Requirement files in `requirements/`
- Requirements pasted directly in chat

**Incremental Generation Guard (mandatory):**

- Scan existing generated tests for `// req: REQ-NNN` and `// source: ...`
- Skip scenarios already covered unless user explicitly requests regeneration
- If multiple requirement files exist, do not auto-generate; require explicit selection of one file per run

## Rules Matrix — Which Agent Enforces What

| Convention                                        | Enforcer                          | Consequences                                                                                                                                                        |
| ------------------------------------------------- | --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Fixture usage (`test` from fixture)               | Generator, SourceCodeGenerator    | Generated tests won't import correctly if violated                                                                                                                  |
| Page object usage (no raw `page.*`)               | Generator, SourceCodeGenerator    | Breaks POM pattern, makes tests brittle                                                                                                                             |
| `test.step()` wrapping                            | Generator, SourceCodeGenerator    | Tests lack step-level granularity for debugging                                                                                                                     |
| `@Step()` on new page object methods              | SourceCodeGenerator               | Methods won't appear as named steps in trace viewer / HTML report                                                                                                   |
| Tag application                                   | Generator, SourceCodeGenerator    | Failed REQ coverage checks; unclear test priority                                                                                                                   |
| Mock data usage                                   | Generator, SourceCodeGenerator    | Hardcoded strings make tests fragile; violate DRY                                                                                                                   |
| REQ traceability (`// req:`)                      | Generator, SourceCodeGenerator, Validator | Fails `validate:generated` quality gate; use `src-derived` when no REQ-NNN found                                                                           |
| `src-derived` traceability                        | SourceCodeGenerator               | Source-derived tests use `// req: src-derived` — must still pass validator                                                                                          |
| Selector strategy (getByRole > getByTestId > CSS) | Healer, SourceCodeGenerator       | Healer must fix page objects first; CSS selectors = brittle tests                                                                                                   |
| No deprecated waits                               | Healer, Generator, SourceCodeGenerator | Flaky tests, race conditions                                                                                                                                   |
| Heal limit (3 max)                                | Healer                            | Prevents over-healing; triggers regeneration                                                                                                                        |
| **Assertion changes require approval**            | Healer                            | ANY `expect()` change → `test.fixme()` + block until human approves; `record_heal_event` rejects `healType="assertion"` without `humanApprovedAssertionChange=true` |
| Assertion count must not drop                     | Healer, Validator                 | `validate_generated_test` + `validate:generated` fail if `expect()` count falls below baseline                                                                      |
| `record_heal_event` requires evidence             | Healer                            | Enforced by MCP schema — `healType`, `evidence`, `productBehaviorChanged` are required fields                                                                       |
| Skip already-covered scenarios                    | SourceCodeGenerator               | Scans `src/tests/ui/generated/` for existing `// source:` headers; avoids duplicate spec files                                                                      |
| Export new page objects from `index.ts`           | SourceCodeGenerator               | Breaks barrel imports; new page object inaccessible to other tests                                                                                                  |

## How to Add a New Agent

If you need a new agent (e.g., API tester, component test planner):

1. **Create `.github/agents/<new-agent-name>.agent.md`**

   - Declare its role (what problem does it solve?)
   - List tools it needs (from `playwright-test`, `playwright-qa-context`, or custom servers)
   - Define its system prompt with clear workflow

2. **Declare startup sequence**

   - What tools must it call first (e.g., framework conventions)?
   - In what order?
   - Document as "non-negotiable"

3. **Define rules** it enforces (or inherits from parent agents)

   - What conventions must downstream agents respect?
   - How does this agent validate inputs from upstream?

4. **Document integration points**

   - What agent calls it (if any)?
   - What output does it produce?
   - Where is that output stored?

5. **Update this file**
   - Add a section under "Agent Roles & Responsibilities"
   - Update the Rules Matrix
   - Add to the Rules Audit section below

## Rules Audit Checklist

Before merging a new agent or updating an existing one:

- [ ] Startup sequence is documented and non-negotiable
- [ ] Enforced conventions are listed explicitly (not implied)
- [ ] Each convention has a "why" (link to project pattern or rationale)
- [ ] Conflicts with existing agents are identified and resolved
- [ ] Integration with downstream agents is clear
- [ ] Output format is specified (file path, structure, required headers)
- [ ] Error handling is defined (what if the app doesn't have a fixture?)
- [ ] Guard rails are in place (e.g., heal limit, over-generation limit)

## Conflict Detection

**When rules might conflict:**

1. **Planner says "E2E tests use this flow" but app is missing that flow**

   - Planner must flag as a gap, not generate a scenario for it
   - Orchestrator stops at Checkpoint 1 so user can decide

2. **Generator creates a page object method that doesn't exist**

   - Generator must add it to the page object file or fail gracefully
   - Rule: Page object extension is allowed; raw `page.*` in test is not

3. **Healer finds a broken selector but test has been healed 3 times already**

   - Healer must STOP and recommend regeneration
   - Do not override the heal limit for convenience

4. **Generator's convention says "no hardcoded strings" but mock data doesn't have the required data**
   - Generator must add it to `shared/mock-data` before using it
   - Better to expand mock data than to hardcode in test

## Version Alignment

Agents are versioned by the Playwright tools they depend on:

- **Playwright Test version → `npx playwright run-test-mcp-server` tools**
- **Custom MCP server version → `playwright-qa-context` tools**

On upgrade:

1. Check Playwright release notes for new MCP tools
2. Update agent `tools:` lists if adopting new capabilities
3. Update system prompts if tool behavior changed
4. Test new agent against a sample app before committing

Record the Playwright version in each agent's comments:

```yaml
# playwright-test-generator.agent.md
# Tested with: Playwright @1.45.0+
# Custom MCP: playwright-qa-context@1.0.0
```
